# 4. 通用功能规则

## 4.1 多租户规则

### 功能说明

系统所有核心业务数据必须进行租户隔离。租户按企业划分，未绑定企业的外部注册用户自动创建个人租户。

### 功能规则

| 编号         | 规则                                            |
| ---------- | --------------------------------------------- |
| TENANT-001 | 所有业务表必须包含 tenant_id，平台级配置除外                   |
| TENANT-002 | 普通用户只能访问本人租户内授权数据                             |
| TENANT-003 | 外部注册用户未加入企业时，系统自动创建个人租户                       |
| TENANT-004 | 公开数据允许跨租户访问，但必须显式设置 is_public=true            |
| TENANT-005 | 管理员访问数据时必须同时校验角色权限和数据范围                       |
| TENANT-006 | 智能问数生成 SQL 时必须自动注入 tenant_id 或 public_flag 条件 |
| TENANT-007 | 租户停用后，该租户下用户不可登录，后台任务暂停执行                     |
| TENANT-008 | 平台超级管理员可查看所有租户，但操作仍需记录日志                      |

### 异常处理

| 场景                | 系统处理             |
| ----------------- | ---------------- |
| 用户访问其他租户私有数据      | 返回 403，无权限       |
| 用户租户被停用           | 登录失败，提示“当前租户已停用” |
| 数据缺失 tenant_id    | 后端拒绝写入，记录异常日志    |
| 公开数据未设置 is_public | 默认视为私有数据         |

---

## 4.2 权限规则

### 功能说明

系统采用 RBAC 权限模型，同时支持菜单权限、按钮权限和数据权限。

### 权限层级

1. 用户。
2. 角色。
3. 菜单权限。
4. 按钮权限。
5. 数据权限。
6. 租户权限。
7. 组织架构权限。

### 权限规则

| 编号        | 规则                          |
| --------- | --------------------------- |
| AUTHZ-001 | 前端菜单由后端返回的权限菜单动态生成          |
| AUTHZ-002 | 前端按钮根据按钮权限控制展示和禁用           |
| AUTHZ-003 | 后端必须对每个接口进行权限校验，不能只依赖前端     |
| AUTHZ-004 | 数据权限应在查询层统一注入               |
| AUTHZ-005 | 平台超级管理员拥有全平台权限              |
| AUTHZ-006 | 租户管理员默认拥有本租户管理权限            |
| AUTHZ-007 | 普通管理员只拥有被分配的模块权限            |
| AUTHZ-008 | 普通用户不可访问管理端路由               |
| AUTHZ-009 | 用户角色变更后，应在下次请求或刷新 Token 时生效 |
| AUTHZ-010 | 权限变更必须记录操作日志                |

### 数据权限类型

| 类型                              | 说明      |
| ------------------------------- | ------- |
| all_platform                    | 全平台     |
| current_tenant                  | 当前租户    |
| current_department              | 当前部门    |
| current_department_and_children | 当前部门及下级 |
| self                            | 仅本人     |
| custom                          | 自定义范围   |

### 通用状态

| 状态       | 说明      |
| -------- | ------- |
| enabled  | 启用      |
| disabled | 停用      |
| deleted  | 已删除，软删除 |
| pending  | 待处理     |
| approved | 已通过     |
| rejected | 已拒绝     |
| running  | 运行中     |
| success  | 成功      |

### 软删除规则

1. 用户、租户、模型、技能、数字员工、工作流、采集任务等核心对象默认采用软删除。
2. 软删除对象不可在普通列表中展示。
3. 管理员可在回收站或审计场景中查看软删除数据。
4. 删除操作必须记录操作日志。

# 5. UF-01 认证与账号功能

## 5.1 注册

### 功能描述

用户通过邮箱或手机号注册账号。用户可选填写企业邀请码，加入企业租户；未填写邀请码时，系统为其创建个人独立租户。

### 页面入口

1. 登录页。
2. 注册页。
3. 邀请注册链接。

### 输入字段

| 字段     | 必填 | 校验规则             |
| ------ | -- | ---------------- |
| 邮箱/手机号 | 是  | 邮箱或手机号格式合法，系统内唯一 |
| 密码     | 是  | 至少 8 位，包含字母和数字   |
| 确认密码   | 是  | 必须与密码一致          |
| 昵称     | 是  | 2-30 个字符         |
| 企业邀请码  | 否  | 如果填写，必须有效且未过期    |
| 验证码    | 是  | 必须正确且未过期         |

### 功能规则

| 编号      | 规则                      |
| ------- | ----------------------- |
| REG-001 | 注册账号必须唯一                |
| REG-002 | 企业邀请码有效时，用户加入对应企业租户     |
| REG-003 | 无企业邀请码时，创建个人租户          |
| REG-004 | 注册成功后默认分配普通用户角色         |
| REG-005 | 注册成功后自动登录或跳转登录页，由系统配置决定 |
| REG-006 | 注册行为记录操作日志              |

### 异常处理

| 场景      | 提示                |
| ------- | ----------------- |
| 账号已存在   | 该账号已注册            |
| 验证码错误   | 验证码错误或已过期         |
| 企业邀请码无效 | 企业邀请码无效或已过期       |
| 密码不符合规则 | 密码需至少 8 位且包含字母和数字 |

前端启动后调用当前用户接口，获取用户信息、租户信息、角色、权限、菜单和系统配置。

4. 菜单权限。

6. 数据权限摘要。

# 32. 后端功能要求

## 32.1 API 通用要求

### 请求要求

1. 所有接口使用 `/api` 前缀。
2. 管理端接口可使用 `/api/admin` 前缀。
3. WebSocket 使用 `/ws` 前缀。
4. 接口默认返回 JSON。
5. 需要登录的接口必须携带 Token。
6. 所有写操作必须校验 CSRF 或等效安全策略，具体实现可按前后端架构决定。

### 响应格式建议

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "request_id": "string"
}
```

### 分页格式建议

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20
}
```

### 错误码建议

| 错误码 | 说明            |
| --- | ------------- |
| 0   | 成功            |
| 400 | 请求参数错误        |
| 401 | 未登录或 Token 无效 |
| 403 | 无权限           |
| 404 | 资源不存在         |
| 409 | 数据冲突          |
| 422 | 业务校验失败        |
| 429 | 请求过于频繁        |
| 500 | 系统异常          |

# 36. 研发实现建议

## 36.1 前端实现建议

1. 使用 React Router 管理路由。
2. 使用后端返回菜单动态生成路由。
3. 使用 shadcn/ui 构建基础组件。
4. 使用 TanStack Query 管理服务端状态。
5. 使用 Zustand 或 Redux 管理全局状态。
6. 使用 WebSocket 管理实时消息。
7. 使用 React Flow 实现工作流画布。
8. 使用 ECharts 或 Recharts 实现图表。
9. 使用响应式布局适配电脑和手机。
10. 大屏组件建议抽象为统一 Widget 模型。

## 36.2 后端实现建议

1. 使用 FastAPI 构建 REST API。
2. 使用 Pydantic 定义请求和响应模型。
3. 使用 SQLAlchemy 或 SQLModel 访问 MySQL。
4. 使用 Alembic 管理数据库迁移。
5. 使用 JWT 实现认证。
6. 使用统一依赖注入进行权限校验。
7. 使用后台任务队列处理采集、清洗、分析、索引。
8. 使用 WebSocket 处理即时通讯。
9. 使用统一日志中间件记录操作日志。
10. 使用服务层统一注入 tenant_id 和数据权限。

## 36.3 模块边界建议

1. 认证、租户、RBAC 是基础模块，优先设计清楚。
2. IM、问数、采集、工作流均依赖权限体系。
3. 模型、技能、知识库是 AI 能力基础设施。
4. 数字员工是模型、技能、知识库、工作流的组合能力。
5. 审计模块应作为横切能力，被聊天、采集、问数、技能调用。
