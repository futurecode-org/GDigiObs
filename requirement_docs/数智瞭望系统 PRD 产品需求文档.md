# 2. 项目背景与目标

## 2.1 项目背景

企业在日常经营、内部协作、外部信息监测、数据分析和智能决策过程中，通常存在以下问题：

1. 数据来源分散，网络公开数据、业务数据、用户上传文件、聊天沟通内容无法统一沉淀。
2. 数据采集、清洗、分析、审计流程割裂，缺少可视化配置和闭环管理。
3. 员工需要通过自然语言快速查询数据，但传统 BI 或数据库系统使用门槛较高。
4. 企业内部协作与智能工具分散，单聊、群聊、任务、知识库、AI 问数、数字员工未形成统一入口。
5. 模型、技能、MCP、Function Call、工作流等 AI 能力需要统一管理和授权。
6. 平台需要支持多租户隔离，既能服务企业内部，也能支持外部注册用户独立使用。

## 2.2 产品定位

数智瞭望系统是一个面向企业和外部注册用户的 **内部协同 + AI 问数平台**，集成数据采集、清洗、分析、数智大屏、社交沟通、知识库、智能问数、模型管理、技能管理、数字员工和审计风控能力。

系统通过三端协同实现：

* 用户端：面向非管理员用户，提供个人大屏、社交通讯、知识库、个人智能问数、技能、数字员工和工作流能力。
* 管理端：面向管理员，提供全局数据治理、租户管理、用户管理、权限管理、数据采集分析、模型技能管理、审计风控和系统智能问数。
* 后端：提供多租户、RBAC、数据采集、AI 调用、工作流执行、消息通信、审计、知识库和系统服务能力。

## 2.3 产品目标

### 业务目标

1. 建立统一的数据采集、清洗、分析和审计平台。
2. 建立企业级 AI 问数能力，支持自然语言生成 SQL、结果可视化和导出。
3. 建立统一的社交协作入口，支持好友、群组、单聊、群聊和文件沟通。
4. 建立 AI 能力平台，支持模型、技能、MCP、Function Call 和数字员工统一管理。
5. 建立多租户隔离和 RBAC 权限体系，满足平台级和企业级管理需求。
6. 建立数据安全与内容审计机制，支持聊天消息敏感度审计和采集数据情感审计。

### 产品目标

1. 用户能在一个平台中完成沟通、问数、查看个人数据和使用数字员工。
2. 管理员能配置采集任务、管理模型技能、查看大屏、审计数据和管理用户权限。
3. AI 能力能通过技能、MCP、Function Call 和工作流灵活扩展。
4. 所有数据访问均受租户、角色、权限和数据可见性控制。

# 3. 用户角色与权限边界

## 3.1 用户角色

| 角色      | 说明                                             |
| ------- | ---------------------------------------------- |
| 外部注册用户  | 自主注册，无企业绑定时作为独立租户；可访问公开数据，可创建技能、数字员工、工作流       |
| 内部普通用户  | 归属于某企业租户的非管理员用户；可访问权限范围内数据，使用社交、问数、知识库、数字员工等功能 |
| 普通管理员   | 拥有部分后台模块管理权限，由平台超级管理员或租户管理员授权                  |
| 租户管理员   | 管理本企业租户内的用户、角色、权限、数据、群组、模型、技能、数字员工等            |
| 平台超级管理员 | 管理所有租户、全局配置、平台模型、平台技能、平台审计和系统级数据               |

## 3.2 租户规则

1. 多租户按企业划分。
2. 企业内部用户必须绑定企业租户。
3. 外部注册用户如果未绑定企业，则系统自动为其建立独立租户。
4. 所有核心业务数据必须带 tenant_id。
5. 公开平台采集/分析数据可被外部注册用户查看。
6. 非公开数据必须受租户、角色、权限和数据范围限制。
7. 管理端系统问数也必须受管理员数据权限限制，不能默认查询全平台数据。

## 3.3 外部用户规则

1. 外部用户可以看到设置为公开的平台采集数据和分析数据。
2. 外部用户可以创建自己的技能、数字员工和工作流。
3. 外部用户和内部员工可以互加好友，但必须双方明确同意。
4. 外部用户和内部员工可以加入同一群，但必须符合群加入规则和邀请确认规则。
5. 外部用户默认不能访问企业内部私有数据。

# 6. 用户端详细需求

## 6.1 注册与登录

### 6.1.1 注册

#### 功能说明

用户可通过账号注册系统。外部注册用户如果未绑定企业，则自动成为独立租户。

#### 主要字段

| 字段     | 说明          | 必填 |
| ------ | ----------- | -- |
| 手机号/邮箱 | 登录账号        | 是  |
| 密码     | 登录密码        | 是  |
| 昵称     | 用户显示名称      | 是  |
| 企业邀请码  | 可选，用于加入企业租户 | 否  |
| 验证码    | 邮箱或短信验证码    | 是  |

#### 业务规则

1. 邮箱或手机号必须唯一。
2. 如果填写有效企业邀请码，则用户加入对应企业租户。
3. 如果未填写企业邀请码，则创建个人独立租户。
4. 注册成功后默认角色为普通用户。
5. 注册成功后进入用户端首页。

### 6.1.2 登录

#### 功能说明

用户使用账号密码登录系统。

#### 业务规则

1. 支持邮箱/手机号 + 密码登录。
2. 登录后根据用户权限进入用户端或管理端。
3. 普通用户默认进入用户端。
4. 管理员可进入管理端。
5. 被封禁或禁用用户不可登录。

## 6.4 个人智能问数

### 功能说明

用户可通过自然语言对权限范围内的数据进行问数。系统支持自然语言生成 SQL、查询结构化数据、调用知识库、生成可视化图表和导出结果。

### 可问范围

1. 用户权限范围内的数据。
2. 公开平台采集数据。
3. 用户个人上传的数据源。
4. 用户有权限访问的知识库。
5. 用户有权限访问的聊天记录和任务数据。
6. 用户所属群或组织授权的数据。

### 功能点

1. 输入自然语言问题。
2. 自动识别数据源。
3. 生成 SQL。
4. SQL 安全校验。
5. 查询数据。
6. 生成自然语言解释。
7. 自动生成图表。
8. 支持折线图、柱状图、饼图、指标卡片、表格。
9. 支持结果导出。
10. 支持保存问数记录。
11. 支持收藏常用问题。
12. 支持追问上下文。

### 查询流程

1. 用户输入问题。
2. 系统判断用户权限和可用数据源。
3. 系统识别问题意图。
4. 系统选择 SQL 查询、知识库检索或混合查询。
5. 如果需要 SQL，则生成 SQL。
6. SQL 进入安全校验。
7. 执行查询。
8. 生成图表和文字解释。
9. 返回结果。
10. 保存问数记录和审计日志。

### SQL 安全规则

1. 默认只允许 SELECT 查询。
2. 禁止 DROP、DELETE、UPDATE、INSERT、ALTER 等写操作。
3. 自动追加租户过滤条件。
4. 自动追加数据权限过滤条件。
5. 查询超出权限时拒绝执行。
6. 查询结果需要脱敏时自动脱敏。
7. SQL 和执行结果需记录日志。

# 9. 核心业务流程

## 9.1 用户注册流程

1. 用户进入注册页。
2. 输入邮箱/手机号、密码、昵称。
3. 可选填写企业邀请码。
4. 系统校验验证码。
5. 系统判断是否绑定企业租户。
6. 如果有有效邀请码，加入企业租户。
7. 如果无邀请码，创建个人独立租户。
8. 创建用户。
9. 分配默认普通用户角色。
10. 注册成功，进入用户端。

---

## 9.2 好友申请流程

1. 用户搜索目标用户。
2. 点击添加好友。
3. 填写验证消息。
4. 系统发送好友申请通知。
5. 对方接受或拒绝。
6. 接受后建立双向好友关系。
7. 双方可发起单聊。

---

## 9.3 建群与邀请流程

1. 用户创建群。
2. 设置群名称、头像、简介。
3. 系统创建群，创建人成为群主。
4. 群主邀请好友或用户申请入群。
5. 群主或管理员审批。
6. 用户入群。
7. 系统发送群系统消息。

---

## 9.4 聊天消息发送与审计流程

1. 用户输入消息。
2. 前端提交消息。
3. 后端进行权限校验。
4. 系统进行敏感内容检测。
5. 如果高风险，自动拦截，生成审计记录和告警。
6. 如果中风险，进入人工复核或按策略暂缓发送。
7. 如果低风险或无风险，消息发送成功。
8. 消息写入聊天记录。
9. 接收方收到实时消息和通知。

---

## 9.5 数据采集流程

1. 管理员选择采集平台。
2. 配置采集方式、地址、规则。
3. 设置手动或定时采集。
4. 启动采集任务。
5. 系统获取数据。
6. 保存原文存档。
7. 写入采集日志。
8. 进入清洗流程。
9. 清洗后进入分析流程。
10. 分析结果可进入大屏、问数和审计。

---

## 9.6 数据清洗流程

1. 系统获取待清洗数据。
2. 执行去重。
3. 执行格式标准化。
4. 执行字段映射。
5. 执行实体识别。
6. 执行敏感词过滤。
7. 执行情感分析。
8. 执行分类打标。
9. 执行异常值检测。
10. 保存清洗结果和日志。

---

## 9.7 智能问数流程

1. 用户输入自然语言问题。
2. 系统识别用户身份、租户和权限。
3. 系统确定可用数据源。
4. 系统判断查询方式：SQL、知识库、混合查询。
5. 系统生成 SQL 或检索计划。
6. 系统进行 SQL 安全校验和权限校验。
7. 执行查询。
8. 生成文字解释。
9. 生成图表。
10. 返回结果。
11. 保存问数记录和审计日志。

---

## 9.8 技能创建流程

1. 用户进入技能管理。
2. 选择技能类型：Function Call、MCP、Skill。
3. 填写配置。
4. 系统校验配置。
5. 用户测试技能。
6. 保存技能。
7. 设置可见范围。
8. 技能可被问数、数字员工或工作流调用。
9. 如果设置为公开，可能需要管理员审核。

---

## 9.9 数字员工执行流程

1. 用户创建数字员工。
2. 配置模型、技能、知识库、工作流。
3. 设置触发方式。
4. 触发数字员工。
5. 数字员工执行工作流。
6. 工作流调用模型、技能、知识库和数据服务。
7. 执行结果按用户设置推送。
8. 系统保存执行记录和审计日志。

---

# 14. 非功能需求

## 14.1 安全需求

1. 所有接口必须鉴权。
2. 所有业务数据必须带 tenant_id。
3. 所有数据查询必须校验租户、角色、权限和数据范围。
4. API Key、MCP 环境变量等敏感信息必须加密存储。
5. 自定义代码运行必须隔离执行。
6. 文件上传必须限制类型和大小。
7. 聊天消息必须进入敏感内容审计。
8. SQL 生成必须进行安全校验。
9. 操作日志和审计日志不可被普通用户删除。

## 14.2 响应式设计

1. 用户端和管理端均需要适配电脑和手机。
2. 大屏在手机端可转为卡片式布局。
3. 表格在手机端支持横向滚动或卡片展示。
4. 聊天界面移动端需接近 IM App 体验。
5. 工作流编辑器在手机端可只支持查看和简单编辑，复杂编排建议使用电脑端。

## 14.3 可扩展性

1. 模型 API 类型需支持扩展。
2. 采集平台需支持扩展。
3. 技能类型需支持扩展。
4. 工作流节点类型需支持扩展。
5. 审计规则需支持自定义扩展。
6. 通知渠道预留短信、企业微信、钉钉、飞书、Webhook。

## 14.4 可观测性

1. 采集任务需有运行日志。
2. 清洗任务需有处理日志。
3. 分析任务需有执行日志。
4. 模型调用需记录请求状态、耗时、Token 消耗。
5. 技能调用需记录入参、出参、状态、耗时。
6. 工作流需记录每个节点执行状态。
7. 问数需记录生成 SQL、执行结果和权限校验结果。

# 16. 验收标准

## 16.1 通用验收标准

1. 用户可成功注册、登录和退出。
2. 普通用户无法访问管理端。
3. 管理员只能访问授权菜单和数据。
4. 不同租户的数据默认隔离。
5. 公开数据可被外部用户访问。
6. 所有重要操作均生成操作日志。
7. 所有审计行为均生成审计日志。

## 16.2 智能问数验收标准

1. 用户可用自然语言发起问数。
2. 系统可根据问题生成 SQL。
3. SQL 只能在权限范围内执行。
4. 系统可返回文字解释、图表和表格。
5. 支持折线图、柱状图、饼图、指标卡片和表格。
6. 支持导出问数结果。
7. 问数历史可查询。

## 16.3 社交验收标准

1. 用户可发送和处理好友申请。
2. 用户可创建群、邀请进群、申请进群。
3. 群主可设置管理员、禁言、踢人、转让群主。
4. 单聊和群聊支持文本、图片、文件、语音、视频文件、表情。
5. 支持消息撤回、已读未读、@成员、消息搜索。
6. 图片、PDF、Office 文件可在线预览。
7. 敏感消息可被自动拦截并进入审计记录。

## 16.4 数据采集验收标准

1. 管理员可创建采集任务。
2. 支持 API、RSS、爬虫采集方式。
3. 支持手动采集和定时采集。
4. 采集数据保存原文存档。
5. 采集日志可查看。
6. 采集数据可进入清洗、分析和审计流程。

## 16.5 AI 能力验收标准

1. 管理员可配置 OpenAI、Anthropic、Ollama API 格式模型。
2. 模型可配置能力标签、上下文长度、Token 上限。
3. 用户可创建 Function Call、MCP、Skill 类型技能。
4. MCP 支持 JSON 导入，优先支持 NPX/UVX 配置。
5. Skill 支持 NPX 安装或 Markdown 导入。
6. 数字员工可绑定模型、技能、知识库和工作流。
7. 工作流支持低代码可视化编排和多种触发方式。

# 17. 风险与待确认事项

## 17.1 产品风险

1. 功能范围较大，建议后续在研发排期中拆分优先级。
2. 自定义代码运行、MCP、Skill 导入存在安全风险，需要沙箱隔离。
3. 爬虫采集需注意目标平台规则、频率限制和合规问题。

5. 外部用户与内部用户互通需要明确企业安全策略。
6. Office 在线预览依赖外部 Microsoft Office 预览能力，需确认文件访问方式和网络可达性。

# 20. 总结

数智瞭望系统的核心是将“企业协同、数据智能、AI 问数、智能体执行、数据治理和审计风控”统一到一个多租户平台中。

用户端重点解决普通用户的协同、问数、知识管理和数字员工使用问题；管理端重点解决平台治理、数据采集分析、权限管理、模型技能管理和审计风控问题；后端则需要提供统一的多租户、RBAC、AI 调度、数据处理和实时通信能力。

本 PRD 可作为后续 UI 原型设计、接口详细设计、数据库设计、研发任务拆分和测试用例编写的基础文档。
